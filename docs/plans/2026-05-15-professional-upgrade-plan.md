# FundAI 专业级改造计划

**版本**：v1.0
**日期**：2026-05-15
**定位**：开源本地基金分析决策系统，后续开发桌面端

---

## 一、现状诊断

基于对项目实际代码的全面审查（非文档描述），以下是当前系统与专业级标准之间的核心差距。

### 1.1 实际技术栈（基于代码审查）

| 层面 | 实际使用 | CLAUDE.md 描述 |
|------|---------|---------------|
| 数据库 | SQLite + aiosqlite | PostgreSQL |
| 缓存 | 文件系统缓存（自实现 CacheClient） | Redis |
| 认证 | 无 | JWT + 手机验证码 |
| ORM | SQLAlchemy + declarative_base（已弃用方式） | SQLAlchemy 2.0 |
| 前端 | Nuxt 4 + Vue 3 + Element Plus + ECharts | 同左 |

### 1.2 五大根本性问题

#### 问题一：分析结果不可信

这是最致命的问题。一个投资决策系统如果创建者自己都不敢用，说明核心价值链断裂。

**数据层**：
- 情绪分析智能体的4个数据工具（新闻舆情、资金流向、社交热度、机构观点）在数据源返回 None 时，返回全零/默认值的"假数据"并标记 `ToolResult.ok()`，LLM 在此基础上生成分析结论，属于"幻觉叠加幻觉"
- [market_data.py:74-84](app/agents/tools/market_data.py) `GetNewsSentimentTool` 在无数据时返回 `{"total": 0, "positive": 0, ...}` 并标记成功，LLM 会认为"近期无新闻"而非"数据不可用"
- 同一基金数据被重复获取3-5次：编排器 `_build_context` 获取一次，TechnicalAgent 和 RiskAgent 各自再获取一次，且各次获取结果可能因缓存窗口不同而不一致

**计算层**：
- RSI 使用简单平均法（[technical.py:139-151](app/agents/technical.py)），行业标准是 Wilder 平滑法
- MA 斜率计算有误（[technical_indicators.py:118](app/agents/tools/technical_indicators.py)），`sum(values[-(period+1):-1])` 在数据刚好 period 长度时取到空列表
- Sortino 比率可能返回 `float('inf')`（[risk.py:137](app/agents/risk.py)），JSON 序列化会失败
- 赎回费率用线性缩放推算（[cost.py](app/agents/cost.py)），严重偏离实际
- 申购费率折扣硬编码为0.1倍（[cost.py](app/agents/cost.py)），不同平台折扣差异大

**决策层**：
- 决策智能体 `use_tools=False`（[decision.py](app/agents/decision.py)），完全依赖 LLM 从上下文综合判断，但上下文本身就包含幻觉数据

#### 问题二：UI 像玩具而非专业工具

- 报告格式像博客文章而非专业分析报告（缺少执行摘要、数据溯源、方法论说明）
- 图表类型单一（仅折线图+雷达图），走势图置信区间实现有误（堆叠方式不正确）
- 无数据表格展示（持仓明细、净值历史等核心数据无处查看）
- 无基金对比功能
- 无报告导出功能
- global.css 近2000行，所有页面样式混在一起
- 首页使用 `alert()` 而非 ElMessage
- useFundService 硬编码 localhost:8000，绕过 Nuxt 代理

#### 问题三：工程质量不达标

- 全局异常处理泄露内部错误信息（[main.py:73-88](app/main.py)）
- Tushare 适配器同步调用阻塞事件循环（[tushare_adapter.py](app/data_sources/tushare_adapter.py)）
- 文件缓存并发写入不安全（[cache.py](app/core/cache.py)），`_write_entry` 和 `_save_index` 无文件锁
- analysis.py 1227行，SSE推送+业务逻辑全堆在端点层
- 服务层存在两套并行定义（analysis.ts vs analysis.service.ts），类型不兼容
- 测试覆盖几乎为零（test_core/ 目录为空）
- 多处使用 `print` 替代 `logger`（knowledge_service.py、vector_store_service.py）
- `datetime.utcnow` 在 Python 3.12 中已弃用（fund.py、analysis.py 多处使用）
- LLM 服务 `chat_with_history_stream` 使用 `chunk.choices[0].content` 应为 `chunk.choices[0].delta.content`（流式响应）
- LLM 服务单例模式实现缺陷：`__init__` 每次都执行重置属性（llm_service.py）
- 同步 OpenAI 客户端在异步上下文中阻塞事件循环

#### 问题四：架构设计存在缺陷

- 全局可变状态（`_active_event_queues`、`_active_event_buffers`、`_active_running_agents`）不支持多进程部署
- SSE 重连依赖内存状态，服务重启后丢失
- 暂停功能只是断开 SSE 连接，后端分析仍在继续，非真正暂停
- 配置系统双重来源（.env + config.json），混乱
- 编排器存在两套并行执行逻辑（`run_analysis` 和 `run_analysis_agents`）
- 分页查询使用 `len(result.all())` 而非 `SELECT COUNT(*)`
- EmbeddingService 内存缓存无上限（embedding_service.py）
- RAG 重排序对中文无效（按空格分词）

#### 问题五：功能维度缺失严重

- 无基金对比分析
- 无基金风格分析（九宫格风格箱）
- 无 VaR / 压力测试
- 无持仓变动追踪
- 无 A类/C类份额对比
- 无报告导出
- 无定投策略分析
- 无情景分析

## 二、改造原则

1. **可信度优先**：先让分析结果可信赖，再谈功能丰富度。一个结果可信的简单系统，远胜于功能花哨但结论不可靠的系统
2. **渐进式改造**：分阶段推进，每个阶段完成后系统都可独立运行和发布
3. **桌面端预备**：架构决策需考虑后续 Electron/Tauri 打包的兼容性
4. **开源品质**：代码质量、测试覆盖需达到开源社区认可的标准
5. **保留首页**：首页 Landing Page 的 UI 设计保留，仅修复 bug 和优化体验
6. **基于实际代码**：以代码审查结果为准，而非 CLAUDE.md 等文档描述

## 三、改造阶段规划

### 阶段一：地基加固

**目标**：修复所有工程缺陷，让系统在可靠性层面达到生产级标准。不增加新功能，但确保现有功能正确运行。

#### 1.1 后端工程修复

| 编号 | 改造项 | 当前问题 | 改造方案 |

| 1.1.11 | Sortino 比率修复 | `float('inf')` 导致 JSON 序列化失败 | 下行标准差为零时返回 `None` 或 `0.0`，并在输出中标注"无下行风险" |
| 1.1.12 | CORS 配置收紧 | `allow_methods=["*"]` 和 `allow_headers=["*"]` 过于宽松 | 限制为实际使用的方法和头部 |

### 阶段二：可信度重建

**目标**：修复所有导致分析结果不可信的问题，确保每个指标的计算正确、数据来源可靠、结论有据可依。

#### 2.1 数据层可信度

| 编号 | 改造项 | 当前问题 | 改造方案 |
|------|--------|---------|---------|
| 2.1.1 | 数据预加载层 | 编排器 `_build_context` 和各 Agent 各自获取数据，同一数据被获取3-5次，结果可能不一致 | 在编排器中统一获取基础数据（净值、持仓、费率等），构建共享上下文对象，各 Agent 从共享上下文读取 |
| 2.1.2 | 情绪数据源诚实化 | 4个情绪工具在数据源返回 None 时返回全零默认值并标记 `ToolResult.ok()`，LLM 误以为"无新闻" | 数据不可用时返回 `ToolResult.fail("数据源暂不支持该数据")` 或 `ToolResult.ok(data=None, note="数据不可用")`，LLM 应明确标注"因数据不足无法分析" |
| 2.1.3 | 数据质量校验 | 数据源返回的数据无合理性检查 | 添加校验层：异常值检测（日增长率超过20%标记）、缺失值处理（插值或标注）、时效性检查（持仓数据标注报告期和滞后天数） |
| 2.1.4 | 数据溯源标注 | 报告中的指标无数据来源和时效信息 | 每个指标标注数据来源（Tushare/AKShare）、获取时间、数据滞后天数，在报告中展示 |
| 2.1.5 | 工具结果验证 | 工具返回数据后直接传给 LLM，无合理性检查 | 添加结果验证层：数值范围检查、空值处理、异常标记 |

#### 2.2 计算层可信度

| 编号 | 改造项 | 当前问题 | 改造方案 |
|------|--------|---------|---------|
| 2.2.1 | RSI 算法修正 | 使用简单平均法（[technical.py:139-151](app/agents/technical.py)），行业标准是 Wilder 平滑法 | 改为 Wilder 平滑法：首次用 SMA 初始化，后续用 `prev_avg_gain * (period-1) + current_gain) / period` 递推 |
| 2.2.2 | MA 斜率计算修正 | [technical_indicators.py:118](app/agents/tools/technical_indicators.py) `sum(values[-(period+1):-1])` 在数据刚好 period 长度时取到空列表 | 添加边界条件检查，数据不足时返回 None 或 0 |
| 2.2.3 | 赎回费率获取优化 | 线性缩放推算严重偏离实际 | 优先从数据源获取实际费率阶梯，线性缩放仅作为最后降级方案并在输出中标注"估算值，可能与实际不符" |
| 2.2.4 | 申购费率折扣修正 | 硬编码0.1倍折扣 | 改为从数据源获取折扣信息，或提供多平台折扣选项（天天基金1折、支付宝1折、银行4折等），让用户选择 |
| 2.2.5 | 计算函数独立测试 | 所有技术指标和风险指标的计算函数无参数化单元测试 | 为每个计算函数编写测试用例，使用已知结果（如通达信/同花顺的计算结果）对比验证 |
| 2.2.6 | 风险指标计算统一 | TechnicalAgent 和 RiskAgent 各自独立计算技术指标，代码重复且可能不一致 | 抽取公共计算模块 `app/core/calculations/`，包含 ma.py、macd.py、rsi.py、volatility.py 等，各 Agent 和工具统一调用 |

#### 2.3 智能体可信度

| 编号 | 改造项 | 当前问题 | 改造方案 |
|------|--------|---------|---------|
| 2.3.1 | 智能体自省机制 | 智能体输出无置信度和数据充足度评估 | 每个智能体输出时附带"置信度"（1-5分）和"数据充足度"（完整/部分/不足），数据不足时明确标注而非猜测 |
| 2.3.2 | 幻觉防护 | 工具返回 None 或数据不足时，LLM 仍编造分析结论 | 修改各智能体的 system prompt，明确要求"数据不足时必须声明无法分析，不得编造结论"；在 BaseAgent 中添加输出校验 |
| 2.3.3 | 工具调用超时与降级 | 每次工具调用无超时控制，可能无限挂起 | 为每次工具调用添加30秒超时，超时后降级为基于已有数据的简化分析并标注 |
| 2.3.4 | LLM 输出结构化约束 | `parse_llm_output` 通过正则和括号匹配提取 JSON，解析脆弱 | 使用 JSON Schema 或 response_format 约束 LLM 输出格式，减少解析失败；添加输出格式校验 |
| 2.3.5 | 情绪智能体降级方案 | 情绪分析数据源全部不可用 | 两个方案择一：(A) 暂时移除情绪分析智能体，待数据源可靠后恢复；(B) 保留但明确标注"情绪分析数据暂不可用，以下分析仅供参考"，降低情绪评分在决策中的权重 |

### 阶段三：专业能力构建

**目标**：补齐专业基金分析系统的核心功能，使分析维度和深度达到行业认可的标准。

#### 3.1 分析维度增强

| 编号 | 改造项 | 说明 | 优先级 |
|------|--------|------|--------|
| 3.1.1 | 基金风格分析 | 九宫格风格箱（大盘/中盘/小盘 x 价值/平衡/成长），晨星核心评级方法。基于持仓市值和估值指标计算 | P0 |
| 3.1.2 | VaR 和压力测试 | 历史模拟法计算 VaR（95%/99%置信度），极端市场情景（2015股灾、2020疫情、2022回调）下的损失模拟 | P0 |
| 3.1.3 | 基金对比分析 | 同类基金横向对比（最多3只），对比维度：收益、风险、费率、持仓、评分 | P0 |
| 3.1.4 | 持仓变动追踪 | 季度间持仓变动分析（换手率、新增/退出重仓股、调仓方向、行业配置变化） | P1 |
| 3.1.5 | 布林带和 KDJ | 补全技术分析指标体系。布林带判断波动率区间，KDJ 补充超买超卖判断 | P1 |
| 3.1.6 | 下行风险和 CVaR | 下行波动率、CVaR（条件在险价值），风险分析精细化 | P1 |
| 3.1.7 | A类/C类份额对比 | 基于持有期计算最优份额类别，考虑申购费+赎回费+销售服务费的总成本 | P1 |
| 3.1.8 | 支撑位/阻力位识别 | 基于历史净值的关键价位识别，辅助止盈止损设置 | P1 |
| 3.1.9 | 成交量分析 | 量价配合分析（缩量/放量判断），需数据源支持 | P2 |
| 3.1.10 | 定投策略分析 | 微笑曲线、平均成本计算、定投 vs 一次性投入对比 | P2 |
| 3.1.11 | 情景分析 | 牛市/熊市/震荡市不同情景下的历史表现和预测 | P2 |
| 3.1.12 | 基金经理稳定性评估 | 变更频率、任职期限、管理规模变化 | P2 |
| 3.1.13 | 基金公司实力评估 | 管理规模、产品线完整度、同类产品平均业绩 | P2 |

#### 3.2 智能体系统增强

| 编号 | 改造项 | 说明 |
|------|--------|------|
| 3.2.1 | 智能体辩论机制 | 分析智能体完成后，进行一轮交叉质疑（如风险分析师质疑技术分析师的走势预测是否充分考虑了风险），决策智能体基于辩论结果做出更审慎的判断 |
| 3.2.2 | 渐进式分析 | 先输出快速概览（10秒内：基于预计算指标的初步结论），再逐步深入分析，用户无需等待完整分析 |
| 3.2.3 | 智能体记忆机制 | 短期记忆（当前会话上下文，已部分实现）+ 长期记忆（历史分析结论和用户偏好，存入本地数据库） |
| 3.2.4 | 编排器统一化 | 移除两套并行执行逻辑（`run_analysis` 和 `run_analysis_agents`），统一为一套基于事件驱动的编排流程 |

## 四、关键决策点

以下决策点需要在实施前明确，每个决策都会影响后续阶段的实施方案。

### 决策1：情绪分析智能体的处理

当前情绪分析的数据源全部返回 None，这是导致分析不可信的重要原因之一。

- **方案A**：接入真实数据源（AKShare 的新闻/资金流向接口），但数据质量可能不稳定
- **方案B**：暂时移除情绪分析智能体，待数据源可靠后再恢复
- **方案C**：保留但降级，明确标注"数据不足，分析结果仅供参考"，降低情绪评分在决策中的权重

### 决策2：桌面端框架

- **方案A**：Electron（生态成熟，但体积大 ~200MB、内存占用高）
- **方案B**：Tauri（体积小 ~10MB、性能好、Rust 后端，但生态较新，Python 后端集成需额外处理）
- **方案C**：先不做桌面端，专注 Web 版本质量，桌面端作为远期规划

### 决策3：认证方案

项目定位为本地使用的开源工具，是否需要认证？

- **方案A**：完全不需要（本地工具，单用户，当前状态）
- **方案B**：可选认证（多用户场景，如家庭/小团队共用，未来扩展）
- **方案C**：简化认证（仅本地 token 防止误操作，不设登录页）

### 决策4：改造节奏

- **方案A**：逐阶段串行推进（每阶段完成后发布一个版本）
- **方案B**：阶段一和阶段二并行（工程修复和可信度修复高度关联，同时进行效率更高）
- **方案C**：先集中做阶段一+二，快速达到"可信可用"，再逐步推进后续阶段

### 决策5：基金对比分析的交互方式

- **方案A**：在工作台内新增"对比模式"，选择多只基金后切换到对比视图
- **方案B**：在报告视图内新增"添加对比基金"按钮，在当前报告旁并排展示
- **方案C**：独立的对比页面，从工作台导航进入

## 六、风险与注意事项

### 6.1 改造过程中的兼容性

- 每个阶段完成后必须确保系统可独立运行和发布
- 数据库 schema 变更需要提供迁移脚本（当前使用 Alembic 但未实际使用）
- API 接口变更需保持向后兼容或提供版本号

### 6.2 性能考虑

- SQLite 单文件数据库在高并发场景下性能有限，但本地单用户场景可接受
- 文件缓存在修复并发安全后，本地场景性能可接受
- LLM 调用是最大性能瓶颈，需考虑缓存和降级策略

### 6.3 安全考虑

- API Key 明文存储在 config.json 中，需在 .gitignore 中排除
- 全局异常处理修复后不再泄露内部信息
- 本地工具场景下，认证非必需但可选

### 6.4 数据源稳定性

- Tushare 需要积分和 Token，部分接口有权限限制
- AKShare 免费但接口可能变更，需做好适配
- 两个数据源都可能出现限流，需做好降级处理
